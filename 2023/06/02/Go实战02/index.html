    进阶

    1. 宕机恢复 recover

    Recover 是一个Go语言的内建函数，可以让进入宕机流程中的 goroutine 恢复过来。

    • recover 仅在延迟函数 defer 中有效，在正常的执行过程中，调用 recover 会返回 nil 并且没有其他任何效果
    • 如果当前的 goroutine 陷入panic，调用 recover 可以捕获到 panic 的输入值，并且恢复正常的执行

    提示：

    1. 在 panic 触发的 defer 函数内，可以继续调用 panic，进一步将错误外抛，直到程序整体崩溃。
    2. 如果想在捕获错误时设置当前函数的返回值，可以对返回值使用命名返回值方式直接进行设置。

    第一个参数的值是这个变量的指针，第二个参数是这个变量的旧值，第三个参数指的是这个变量的新值。

    原子操作与互斥锁的区别

    • 互斥锁是一种数据结构，用来让一个线程执行程序的关键部分，完成互斥的多个操作。
    • 原子操作是针对某个值的单个互斥操作。

    可以把互斥锁理解为悲观锁，共享资源每次只给一个线程使用，其它线程阻塞，用完后再把资源转让给其它线程。

    CAS操作的做法趋于乐观，总是假设被操作值未曾被改变（即与旧值相等），并一旦确认这个假设的真实性就立即进行值替换，那么在被操作值被频繁变更的情况下，CAS操作并不那么容易成功。

    atomic包提供了底层的原子性内存原语，这对于同步算法的实现很有用。这些函数一定要非常小心地使用，使用不当反而会增加系统资源的开销，对于应用层来说，最好使用通道或sync包中提供的功能来完成同步操作。

    8. 定时器

    NewTicker

    NewTicker() 启动后，会自己维护一个过期时间的通道（Channel）也就是代码里的 <-ticker.C
    这句意思就是时间一到，ticker会通过管道发出一个信号给CPU，告诉CPU时间到了，该执行定时里的方法了，信号发出后，ticker会自动的重置定时的剩余时间，然后再进行下一轮的发送信号执行方法

    NewTimer

    NewTimer()相当于一个睡眠（延时执行）。

    时间一到，timer会通过管道发出一个信号告诉CPU该执行定时里的代码了，然后这个管道就销毁了
    除非使用 timer.Reset(time.Duration * 2) 来重新激活这根管道，让它重置定时的剩余时间，到下一轮定信号发出后，还要再次重置

    注意：定时任务执行时间过长且超过定时的间隔时间时，定时的间隔时间到了之后会等待定时任务执行完才会进行下一轮的定时任务.

    终止与重置：

    • 如果计时器已经过期，并且 t.C 已经被读完了，那么可以直接使用 Reset。
    • 而如果程序 Reset 之前未从 t.C 中读取过值的话，就需要调用 Stop 来结束定时器，才能使用 reset。 输出： 因为 time.sleep()是让主协程睡大觉，而 timer.C 读的那条管道的协程是独立的。

    为什么空结构体的宽度是0？
    Go 编译器在内存分配时做的优化项，当发现 size 为 0 时，会直接返回变量 zerobase 的引用，该变量是所有 0 字节的基准地址，不占据任何宽度。

    使用场景

    空结构体常作为占位符使用

    1.实现方法接收者

    2.实现集合类型

    但有个问题，就是集合类型的使用，只需要用到 key（键），不需要 value（值）。

    空结构体作为占位符，不会额外增加不必要的内存开销，很方便的解决了。

    12. 垃圾回收

    常见垃圾回收算法：

    1. 引用计数：每个对象维护一个引用计数，为0时回收。（Python、PHP、Swift）
    2. 标记清除：遍历并标记有引用的对象，没有标记的回收。（Golang）
    3. 分代收集：按照对象生命周期划分代空间，不同代有不同回收算法和频率。（Java）

    三色标记法

    golang 的垃圾回收采用的是 标记-清理（Mark-and-Sweep） 算法就是先标记出需要回收的内存对象，然后清理，使用就是三色标记法。

    1.三色标记：

    • 白色对象 — 潜在的垃圾，其内存可能会被垃圾收集器回收；
    • 黑色对象 — 活跃的对象，包括不存在任何引用外部指针的对象以及从根对象可达的对象；
    • 灰色对象 — 活跃的对象，因为存在指向白色对象的外部指针，垃圾收集器会扫描这些对象的子对象；

    2.根对象：垃圾回收中最先检测的对象

    • 全部变量
    • 执行栈：每个 goroutine 都包含自己的执行栈，这些执行栈上包含栈上的变量及指向分配的堆内存区块的指针。
    • 寄存器：寄存器的值可能表示一个指针，参与计算的这些指针可能指向某些赋值器分配的堆内存区块。

    3.工作原理

    • 在垃圾收集器开始工作时，程序中不存在任何的黑色对象
    • 垃圾收集的根对象会被标记成灰色
    • 垃圾收集器只会从灰色对象集合中取出对象开始扫描
    • 当灰色集合中不存在任何对象时，标记阶段就会结束

    4.GC触发时机
    Go 语言中对 GC 的触发时机存在两种形式：

    • 主动触发，通过调用 runtime.GC 来触发 GC，此调用阻塞式地等待当前 GC 运行完毕。
    • 被动触发，分为两种方式：
      • 使用系统监控，当超过两分钟没有产生任何 GC 时，强制触发 GC。
      • 使用步调（Pacing）算法，其核心思想是控制内存增长的比例。

    内存泄漏

    根本原因：预期能被很快释放的内存由于附着在了长期存活的内存上，或生命期意外的被延长，导致预计能够立即回收的内存长时间得不到回收

    示例：
    当有一个全局对象时，可能不经意间将某个变量附着在其上，且忽略将其进行释放，则该内存永远不会得到释放

    goroutine泄漏

    goroutine需要消耗内存维护执行代码的上下文信息，这些内存不会释放。

    1.如果一个程序持续不断的产生新的goroutine，且不结束已创建的goroutine，并复用内存，就会造成内存泄漏。

    2.如果一个channel尝试向一个没有接收方，无缓冲的channel发消息，该goroutine会被永久休眠，整个goroutine和执行栈都不会释放

    切片内存泄漏

    在 Go 语言中，当一个切片被创建后，它会引用一个底层数组，该数组的容量可能大于实际使用的长度。
    当切片中的元素不再需要时，如果没有正确地管理切片的长度和容量，将可能导致内存泄漏问题。

    常见原因

    1 没有及时释放不再使用的切片
    如果一个切片在使用完毕后没有及时调用 slice = nil 或者 slice = []type{} 来释放内存，那么它所占用的内存就会一直存在，从而导致内存泄露。

    2 没有使用切片的截断功能
    在使用切片时，如果没有使用切片的截断功能来裁剪底层数组中不需要的部分，那么就会导致内存泄露。
    因为切片底层是一个数组，如果我们只使用了切片的一部分，而没有将其截断，那么底层数组中未使用的部分仍然会被保留在内存中，从而导致内存泄漏。

    3 在循环中使用 append 函数
    在循环中使用 append 函数时，如果没有提前将切片的容量设置足够大，那么 append 函数可能会导致底层数组重新分配空间，从而导致内存泄露。

    解决方案
    1. 在不使用切片时，及时使用 nil 进行释放
    2. 在使用切片时，合理设置其容量，以避免不必要的内存重新分配。
      示例： s没有设置其容量，在切片需要重新分配内存空间时，旧的内存空间无法被及时释放，从而导致内存泄露问题. 使用 make 函数创建了一个容量为 10 的切片 s，避免了不必要的内存重新分配，从而避免了内存泄露问题。

    13. context

    context是go中用来进程通信的一种方式，底层是借助channel与snyc.Mutex实现，用于管理请求作用域数据和控制请求的生命周期。

    基本介绍

    一个接口：Context，规定了context的4个基本方法
    四种实现：

    • emptyCtx 实现了一个空context，可用作根节点
    • cancelCtx 实现了一个带cancel功能的centext，可主动取消
    • timerCtx 实现了一个通过定时器timer和截止时间deadline定时取消的context
    • valueCtx 实现了一个可以通过key，value来存数据的context
      六个方法：
    • Background 返回一个emptyCtx作为根节点
    • TODO 返回一个emptyCtx作为未知节点
    • WitchCancel 返回cancelCtx
    • WithDeadline
    • WithTimeout
    • WithValue

    源码分析

    1.Context接口

    • Deadline() ：返回一个time.Time，表示当前Context应该结束的时间，ok则表示有结束时间
    • Done()：返回一个只读chan，如果可以从该 chan 中读取到数据，则说明 ctx 被取消了
    • Err()：返回 Context 被取消的原因
    • Value(key)：返回key对应的value，是协程安全的

    2.创建一个有截止时间的Context：时间到期时，会自动取消相关工作

    使用建议

    1. 不要将 Context 塞到结构体里。直接将 Context 类型作为函数的第一参数，而且一般都命名为 ctx
    2. 不要向函数传入一个 nil 的 context，如果你实在不知道传什么，标准库给你准备好了一个 context：todo
    3. 不要把本应该作为函数参数的类型塞到 context 中，context 存储的应该是一些共同的数据。例如：登陆的 session、cookie 等

    14. 内存逃逸分析

    内存逃逸分析是go的编译器在编译期间，根据变量的类型和作用域，确定变量在堆上还是栈上。
    如果变量需要分配在堆上，则称为内存逃逸了。

    概念

    1.为什么需要内存逃逸分析？
    因为go语言是有GC自动内存管理的，开发者不需要关心内存释放问题。这样编译器和运行时(runtime)就需要准确分配和管理内存，在编译期间确定变量放在堆还是栈空间。

    2.变量放错位置会怎样？
    栈空间生命周期是和函数生命周期相关的，函数的局部变量离开函数范围就会失效（如函数结束），所以这样的变量需要放在堆空间。

    3.堆和栈的使用成本

    • 堆空间使用成本高：堆内存要向系统申请和归还
    • 栈内存在程序运行时由程序自己确定，且分配和回收成本低，只需2个CPU指令：PUSH分配空间、POP释放空间。

    场景

    Go逃逸分析工具：编译时加-gcflags=-m

    1.return局部变量指针

    2.返回值

    在函数中定义一个变量并将其返回时，该变量就会被分配到堆上，因为栈上的数据只能在当前函数的范围内使用，一旦退出函数，栈上的数据就会被销毁，并且无法再访问该变量的值。

    3.interface{}
    当函数变量类型是interface{}时，因为编译器无法推断运行时变量实际类型，所以会发生逃逸。

    4.栈空间不足
    因为栈的空间是有限的，所以在分配大块内存时，会考虑栈空间内否存下，如果栈空间存不下，会分配到堆上。

    在 Go 语言中，以下情况会将变量分配到堆上：

    1. 在函数中使用 new 或 make 创建的变量。这些变量的底层数据结构需要分配在堆上，并由 GC 进行管理。
    2. 函数返回一个指针类型的变量。如果返回的变量存在于调用函数之外，那么就需要在堆上进行分配，以保证变量的可访问性。
    3. 变量的大小超过了一定阈值。如果一个变量的大小超过了编译器默认的阈值（通常为 32 bytes），则该变量可能会被分配到堆上。

    性能

    操作栈空间比堆空间要快多了，而且使用堆空间还会有GC问题，频繁的创建和释放堆空间，会增加GC的压力
