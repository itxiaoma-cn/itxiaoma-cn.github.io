    控制节点组件

    • APIServer：整个集群的控制中枢，提供集群各个模块间的数据交换，并将集群状态和信息存储到Etcd中。同时也是集群管理、资源配额、安全机制的入口，为集群各类资源对象提供增删改查及watch接口。
    • Scheduler：集群调度中心，通过调度算法将Pod分配到最佳Node上。
    • Controller Manager：集群状态管理器，配合Scheduler保证Pod副本数量。
    • Etcd：分布式键值数据存储组件，作为k8s集群的持久化存储系统。

    组件状态

    • APIServer：无状态服务，和节点一对一对应
    • Scheduler：有状态
    • Controller Manager：有状态
    • Etcd：无状态

    工作节点组件

    • Kubelet：与Master通信，管理Pod，对容器进行健康检查及监控，同时上报节点和节点Pod状态。
    • Kube-Proxy：负责Pod间通信和负载均衡
    • Runtime（Docker/Containerd）：容器管理
    • CoreDNS：集群内部Service解析，把Service名称解析成ServiceIP，通过ServiceIP连接到应用上
    • Calico：符合CNI标准的网络插件，给每个Pod分配一个不会重复的IP地址，并把每个节点当做路由器，让Pod可以通过IP访问到其他Pod

    Pod

    Pod是Kubernetes中最小的可部署单元，它是一个由一个或多个容器组成的集合。这些容器共享相同的网络命名空间、存储卷和其他资源，并在同一主机上运行。

    Pause：每个Pid包含一个Pause容器，负责僵尸进程回收，Pod内容器共享存储、网络、PID、IPC等。

    为什么要引入Pod

    1. 强依赖的服务需要部署在一起（如Nginx+PHP）
    2. 多个服务需要协同工作（如需要初始化数据等）
    3. 兼容其他CRI标准的运行时

    Pod状态

    状态 说明
    Pending Pod已被Kubernetes系统接收，但仍有一个或多个容器未被创建。可以使用kubectl describe命令查看处于Pending状态的原因。
    Running Pod已经被绑定到一个节点上，并且所有的容器都已经被创建。至少有一个容器处于运行状态，或者正在启动或重启。可以使用kubectl logs命令查看Pod的日志。
    Succeeded 所有容器执行成功并终止，并且不会再次重启。可以使用kubectl logs命令查看Pod的日志。
    Failed 所有容器都已终止，并且至少有一个容器以失败的方式终止。即容器以非零状态退出或被系统终止。可以使用kubectl logs和kubectl describe命令查看Pod的日志和状态。
    Unknown 通常是由于通信问题造成无法获得Pod的状态。
    ImagePullBackOff、ErrImagePull 镜像拉取失败，可能是由于镜像不存在、网络不通或需要登录认证引起的。可以使用kubectl describe命令查看具体原因。
    CrashLoopBackOff 容器启动失败，可以使用kubectl logs命令查看具体原因。一般原因包括启动命令不正确、健康检查不通过等。
    OOMKilled 容器内存溢出，可能是容器的内存限制设置过小或程序本身存在内存溢出。可以使用kubectl logs命令查看程序启动日志。
    Terminating Pod正在被删除，可以使用kubectl describe命令查看状态。
    SysctlForbidden Pod自定义了内核配置，但kubelet没有添加内核配置或配置的内核参数不支持。可以使用kubectl describe命令查看具体原因。
    Completed 容器内部主进程退出，一般用于计划任务执行结束时显示该状态。此时可以使用kubectl logs查看容器日志。
    ContainerCreating Pod正在创建，一般是由于正在下载镜像或存在配置问题。可以使用kubectl describe命令查看具体原因。

    • Always 总是拉取，当镜像 tag 为 latest 时，且 imagePullPolicy 未配置，默认为 Always

    • IfNotPresent 镜像不存在时拉取镜像，如果 tag 为非 latest，且 imagePullPolicy 未配置，默认IfNotPresent

    Pod探针

    startupProbe

    Kubernetes1.16 新加的探测方式，用于判断容器内的应用程序是否已经启动。

    如果配置了 startupProbe，就会先禁用其他探测，直到它成功为止。

    如果探测失败，Kubelet会杀死容器，之后根据重启策略进行处理，如果探测成功，或没有配置 startupProbe，则状态为成功，之后就不再探测。

    livenessProbe

    用于探测容器是否在运行，如果探测失败，kubelet 会“杀死”容器并根据重启策略进行相应的处理。如果未指定该探针，将默认为 Success

    readinessProbe

    一般用于探测容器内的程序是否健康，即判断容器是否为就绪（Ready）状态。

    如果是，则可以处理请求，反之 Endpoints Controller 将从所有的 Service 的 Endpoints中删除此容器所在 Pod 的 IP 地址。如果未指定，将默认为 Success

    探针实现方式

    • ExecAction 在容器内执行一个指定的命令，如果命令返回值为 0，则认为容器健康
    • TCPSocketAction 通过 TCP 连接检查容器指定的端口，如果端口开放，则认为容器健康
    • HTTPGetAction 对指定的 URL 进行 Get 请求，如果状态码在 200~400 之间，则认为容器健康

    Pod重启策略

    • Always 默认策略。容器失效时，自动重启该容器
    • OnFailure 容器以不为 0 的状态码终止，自动重启该容器
    • Never 无论何种状态，都不会重启

    注意：

    • Endpoint（端点）是一种资源对象，用于表示服务的网络终点。它定义了一个服务所关联的一组网络地址，这些地址可以是Pod、Node或其他外部服务的IP地址和端口。

    PreStop：生命周期钩子 ，是容器级别

    preStop 和 postStart

    preStop和postStart是容器的生命周期钩子（lifecycle hook），用于在容器启动前和停止后执行一些操作。

    • preStop钩子定义了容器在终止之前要执行的命令。它可以用来处理容器的清理工作，例如保存状态、关闭连接或发送信号给其他进程。preStop钩子会在容器接收到终止信号（如SIGTERM）时触发，并且在容器终止之前等待一段时间，以便完成预处理操作。

    • postStart钩子定义了容器在启动后要执行的命令。它可以用来进行一些初始化操作，例如加载配置、启动服务或检查依赖项。postStart钩子会在容器成功启动后立即触发。
