    6.新数据类型

    Bitmaps 位操作字符串

    • Bitmaps实际上就是字符串(k-v)，但是可以对字符串进行位操作
    • 可以把Bitmas想象成以位为单位，只能存0和1的数组，数组下标叫做偏移量

    常用命令

    setbit key offset value  //设置值

getbit key offset              //取值

bitcount （start end     //统计数值

bitop and/or/not/xor destkey key1 key2  //复合操作，交并非异或，结果保存在destkey

    测试

    127.0.0.1:6379> setbit k1 1 1
(integer) 0
127.0.0.1:6379> setbit k1 2 1
(integer) 0
127.0.0.1:6379> setbit k1 4 1
(integer) 0
127.0.0.1:6379> getbit k1 1
(integer) 1
127.0.0.1:6379> getbit k1 3
(integer) 0
127.0.0.1:6379> bitcount k1
(integer) 3
127.0.0.1:6379> setbit k2 3 1
(integer) 0
127.0.0.1:6379> bitop and k1_and_k2 k1 k2 //k1和k2都有值 
(integer) 1
127.0.0.1:6379> bitcount k1_and_k2
(integer) 0
127.0.0.1:6379> bitop or k1_or_k2 k1 k2 //k1或k2有值 
(integer) 1
127.0.0.1:6379> bitcount k1_or_k2
(integer) 4

    结论

    但 Bitmaps 并不是万金油， 假如该网站每天的独立访问用户很少， 例如只有 10 万（大量的僵尸用户） ， 那么两者的对比如下表所示， 很显然， 这时候使用 Bitmaps 就不太合适了， 因为基本上大部分位都是 0。

    Bitmaps适合百万级以上数据量存储。

    HyperLogLog

    基数问题

    求集合中不重复元素个数的问题称为基数问题

    解决方案：

    • Mysql使用distinct count计算
    • Redis使用hash，set，bitmaos等数据结构处理

    基数统计

    基数统计就是在误差可接受的范围内，快速计算基数。

    • Redis HyperLogLog是用来做基数统计的算法，优点是输入大量数据时，所需空间固定(很小)。
    • 在 Redis 里面，每个 HyperLogLog 键只需要花费 12 KB 内存，就可以计算接近 2^64 个不同元素的基数。
    • HyperLogLog 只会根据输入元素来计算基数，而不会储存输入元素本身，所以 HyperLogLog 不能像集合那样，返回输入的各个元素。

    常用操作

    pfadd key element //添加元素 基数变化返回1，不变返回0

pfcount key                //统计基数

pfmerge destkey key1 key2  //合并

    Geospatial 经纬度

    Redis3.2新增，提供经纬度设置，查询范围，距离查询等操作

    求集合中不重复元素个数的问题称为基数问题

    解决方案：

    • Mysql使用distinct count计算
    • Redis使用hash，set，bitmaos等数据结构处理

    常用操作

    geoadd key longitude latitude member //设置经纬度

geopos key member                        //获取经纬度

geodist key member1 member2 (m km ft mi) //计算两个位置之间的直线距离

georadius key longitude latitude radius (m km ft mi) //以给定的经纬度为中心，找出某一半径的内元素
